GGUF 与远程仓库 llama.cpp 原生使用 GGUF；本地目录会自动选择最大的 .gguf 文件。
本地路径请指向 .gguf 文件或包含 .gguf 的目录；远程仓库可填 TheBloke/xxx-GGUF:Q4_K_M 这类 llama.cpp 支持的 repo:quant。
llama.cpp 可用 reasoning-format deepseek 把思考放进 reasoning_content；Open WebUI 通常能分离显示，普通客户端可用 none 原样保留。
上下文与显存估算 权重 + KV cache + 运行余量；实际以 llama.cpp 日志为准。
llama.cpp 不传 mmproj 时就是纯文本服务；开启后会在配置和估算里明确不预留视觉 projector。
输入模型后会估算对应上下文长度的显存占用。
模型名里包含 7B、14B、27B 这类规格时估算最准确；长上下文主要增加 KV cache 显存。
仅本机访问会绑定 127.0.0.1；局域网访问会绑定本机 IPv4 地址。
GPU 设备 勾选要暴露给 llama.cpp 的显卡；异构双卡会按可用显存建议 tensor split。

异构双卡优化 为显存和算力不一致的多 GPU 组合生成 llama.cpp split；单 96GB 卡会自动走单卡方案。
选择 GPU 后会显示推荐方案。
layer 模式按层把权重放到不同 GPU，最适合异构双卡；row/tensor 可能提高吞吐，但慢卡更容易拖低单路延迟。main GPU 在启动命令里使用容器内可见序号，管理器会自动映射宿主机 GPU 编号。
本地 Claude 单路优先选 layer；想测试吞吐可以试 row；tensor 是实验项，异构双卡上通常不如 layer 稳。

下载模型

选择预设后仍可手动改模型 ID。 规格越大效果通常越好但更吃显存；量化精度越低越省显存，可能损失一点质量。
