# 本地模型 2 小时综合能力测试任务

> 用法：把“给模型的完整任务”整段复制到 Claude Desktop / Cowork / Claude Code 里，让它使用你的本地 vLLM Claude 兼容桥执行。建议新开一个空项目目录运行，不要给它额外提示。

## 测试目标

这个任务用来测试本地模型在 1.5-2 小时内的综合 agent 能力：

- 是否能理解一个中等复杂需求
- 是否会先建立计划再动手
- 是否能正确使用文件、命令、测试工具
- 是否能处理错误日志和边界情况
- 是否能保持上下文，不丢关键约束
- 是否能在结束时给出可验证结果

## 建议评分

总分 100 分：

- 需求理解与计划：15 分
- 项目结构和代码质量：20 分
- 功能完整度：25 分
- 错误处理和边界情况：15 分
- 测试与自检：15 分
- 最终汇报清晰度：10 分

扣分重点：

- 编造不存在的运行结果
- 没有实际运行测试却说测试通过
- 丢失用户硬性要求
- 破坏工作区无关文件
- 把工具调用写成普通文本而不是实际调用

# 给模型的完整任务

你现在要完成一个本地离线小工具，目标是在 2 小时内交付可运行版本。请在当前工作区创建一个新文件夹：

```text
agent-log-auditor
```

不要修改这个文件夹之外的任何文件。

## 背景

我有很多本地模型、OpenWebUI、Claude Desktop、vLLM、Docker 的日志。它们格式很乱。我需要一个小工具，把一组日志文件整理成一份 Markdown 审计报告，帮助我快速判断：

- 哪些服务启动过
- 哪些模型加载过
- 有哪些错误
- 是否发生过工具调用
- 是否有疑似联网、文件访问、命令执行行为
- 哪些问题需要我继续处理

## 你的任务

请实现一个命令行工具：

```bash
python audit_logs.py --input samples --output report.md
```

要求：

1. 自动读取 `--input` 目录下所有 `.log`、`.txt`、`.jsonl` 文件。
2. 支持递归子目录。
3. 解析日志中的时间、级别、服务名、模型名、错误、工具调用、URL、文件路径、端口。
4. 生成一份 Markdown 报告。
5. 报告必须包含：
   - 总览
   - 服务启动记录
   - 模型加载记录
   - 错误与警告
   - 工具调用与命令行为
   - URL / 网络访问迹象
   - 文件路径访问迹象
   - 端口与本地服务
   - 风险等级
   - 后续处理建议
6. 不能依赖联网。
7. 只能使用 Python 标准库。
8. 需要包含样例日志和测试。

## 必须创建的文件

你必须创建这些文件：

```text
agent-log-auditor/
  audit_logs.py
  README.md
  samples/
    vllm.log
    claude_bridge.log
    openwebui.jsonl
    nested/
      docker.txt
  tests/
    test_audit_logs.py
```

## 样例日志要求

请你自己生成样例日志，至少覆盖：

- vLLM 模型启动成功
- Docker 容器启动
- Claude bridge 请求
- 一个 tool_use
- 一个 tool_result
- 一个报错
- 一个 Windows 路径
- 一个 HTTP URL
- 一个本地端口
- 一个 Hugging Face 模型 ID

## 报告格式要求

生成的 `report.md` 要专业、清晰、适合人读。示例结构：

```md
# 日志审计报告

## 总览

## 关键发现

## 服务启动记录

## 模型加载记录

## 错误与警告

## 工具调用与命令行为

## 网络访问迹象

## 文件路径访问迹象

## 端口与本地服务

## 风险等级

## 后续处理建议
```

## 风险等级规则

请实现一个简单评分系统：

- 出现 `ERROR`、`Traceback`、`failed`：加 2 分
- 出现 shell 命令、tool_use、Bash、PowerShell：加 2 分
- 出现外部 URL：加 1 分
- 出现用户目录、AppData、密钥字样、token 字样：加 2 分
- 出现模型启动失败：加 3 分

风险等级：

- 0-2：低
- 3-5：中
- 6-8：高
- 9+：严重

## 测试要求

必须写测试，至少覆盖：

1. 能递归发现日志文件。
2. 能识别错误。
3. 能识别 URL。
4. 能识别 Windows 路径。
5. 能识别工具调用。
6. 能生成 Markdown 报告。
7. 风险等级计算正确。

测试可以用 Python 标准库 `unittest`，不要用 pytest。

## 工作方式要求

请严格按这个流程：

1. 先简短复述你的理解。
2. 列出 5-8 步计划。
3. 创建文件和代码。
4. 运行测试。
5. 如果测试失败，修复后重跑。
6. 运行一次真实样例生成 `report.md`。
7. 最后汇报：
   - 创建了哪些文件
   - 测试是否通过
   - 如何运行
   - 报告位置
   - 你认为还有哪些改进点

## 重要约束

- 不要访问互联网。
- 不要修改 `agent-log-auditor` 之外的文件。
- 不要删除任何已有文件。
- 不要伪造测试结果。
- 如果命令失败，必须展示失败原因并修复。
- 如果你无法完成某部分，要明确说明。
